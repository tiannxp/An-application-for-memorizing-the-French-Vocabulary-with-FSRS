## 二、当前 MVP 已具备的能力

截至今天收工，阅读器 MVP 已经具备一个最小但完整的闭环：

### 1. 导入闭环

- 可以粘贴网页正文或富文本内容
- 可以保留基础 HTML 结构
- 可以进行去噪处理
- 可以在预览态手动指定“正文从此开始”
- 可以保存为文章

### 2. 阅读闭环

- 可以打开文章
- 可以查看清洗后的阅读排版
- 可以调整字体大小

### 3. 标注闭环

- 可以选中文本创建高亮
- 可以给高亮设置颜色
- 可以给高亮写笔记
- 可以删除高亮

### 4. 词袋闭环

- 可以从高亮进入词袋
- 可以记录表达、上下文、备注
- 可以写入 `vocab_inbox`

### 5. 清理闭环

- 可以进入清理模式
- 可以删除不需要的段落 / 图片模块
- 可以对段落做基础编辑
- 可以保存清理后的版式结果

这意味着，今天的版本已经不是“静态原型”，而是一个真正能跑的 Reader MVP。

## 三、当前版本的不足

虽然今天已经把 MVP 救活，但从产品角度看，当前阅读器仍然停留在“第一阶段可用”。

目前主要不足如下：

- 高亮仍然依赖单文本节点路径，复杂跨段落高亮还不稳定。
- 清理模式还比较偏工程工具感，不够产品化。
- 右侧高亮区目前只有列表和笔记，层次感还不够。
- UI 风格偏基础 Bootstrap，功能够用，但视觉辨识度和高级感不足。
- 导入虽然已经支持结构保留，但对不同网站的适配策略仍然比较粗。
- 还没有“阅读 -> 提炼 -> 写作 -> 制卡”的完整高阶闭环。

## 四、未来产品拓展方向

下面这部分不是随意发散，而是按“最值得做、最容易形成产品壁垒”的方向来整理。

### A. 笔记系统升级

笔记系统未来不应该只是一个普通 textarea，而应该变成“结构化学习笔记系统”。

可以考虑：

- 颜色语义化
  - 黄：生词 / 表达
  - 蓝：写作句式
  - 绿：观点 / 论据
  - 红：错误警示 / 易混点
  - 紫：值得模仿的高级表达
- 给高亮增加标签体系
  - 生词
  - 固定搭配
  - 句式模板
  - 写作素材
  - 论证方式
  - 金句
- 给笔记增加结构化字段
  - 我的理解
  - 中文释义
  - 使用场景
  - 是否值得制卡
  - 是否值得写作复用
- 支持“高亮分组”
  - 按主题分组
  - 按文章章节分组
  - 按用途分组

这会让阅读器逐渐从“看文章工具”升级成“知识加工工具”。

### B. UI 与交互升级

当前 UI 虽然已经可用，但未来完全可以朝更成熟的阅读产品风格推进。

建议方向：

- 视觉层升级
  - 更好的留白和层次
  - 更稳定的阅读字号体系
  - 更柔和但有区分度的高亮颜色
  - 更像专业阅读器的阅读区宽度与边距
- 交互层升级
  - 选中后浮动工具条
  - 高亮 hover 快速编辑
  - 笔记侧栏支持折叠 / 固定
  - 快捷键体系
- 状态层升级
  - 导入中
  - 正在清洗
  - 已保存
  - 未保存修改
  - 后端未连接

如果这部分做好，阅读器会明显从“开发者工具页面”进化为“产品页面”。

### C. AI 接入方向

AI 不应该只是一个“可选外挂”，而应该成为阅读器的增益引擎。

最值得优先做的 AI 能力有：

#### 1. 一键提炼文章大纲

目标：

- 自动生成文章结构图
- 提炼核心观点
- 标出段落层级
- 生成简要摘要

价值：

- 降低阅读门槛
- 帮助快速回顾
- 为写作复用提供结构骨架

#### 2. 一键提取优秀写作句子

目标：

- 自动筛出值得模仿的句子
- 按类别标注：
  - 观点表达
  - 转折句
  - 让步句
  - 总结句
  - 举例句
  - 论证句

价值：

- 直接服务写作训练
- 让阅读器不仅“懂内容”，还“懂表达”

#### 3. 一键提取可制卡表达

目标：

- 自动从文章中标出高价值词组 / 搭配 / 固定表达
- 自动进入候选词袋
- 允许用户批量确认

价值：

- 降低手动摘词成本
- 与闪卡系统天然衔接

#### 4. AI 学习伴读

目标：

- 用中文解释难句
- 对比原句与简化改写
- 解释词组语气和使用语境
- 回答“为什么作者这样写”

价值：

- 形成“阅读教练”体验
- 使阅读器从静态工具变成陪练系统

### D. 与主程序深度接入

这是非常关键的一步。

阅读器未来不应该是独立小页面，而应该是主程序中的一级模块。

建议路线：

- 在主程序中增加 Reader 模块入口
- 使用快捷键 `R` 进入阅读器
- 再按 `R` 退出阅读器 / 返回主界面
- 与闪卡系统共享同一数据源
- 与写作系统共享同一素材源

这样会带来几个明显好处：

- 使用者心智统一
- 减少模块切换成本
- 阅读、高亮、词袋、制卡、写作可以形成一条操作流

## 五、百万年薪产品经理视角下的未来产品路线

如果我们不是只把它当成一个练手页面，而是认真往“高价值学习产品”推进，那么它的未来不应该停在 Reader。

真正有价值的，是围绕“阅读 -> 提炼 -> 记忆 -> 写作 -> 输出”做完整闭环。

### 路线一：阅读器 -> 闪卡闭环

闭环链路：

- 读文章
- 标高亮
- 加词袋
- 批量制卡
- 复习
- 从卡片反跳回原文

这个方向的价值在于：

- 直接提升词汇积累效率
- 构建个性化语言数据库
- 形成长期复利

### 路线二：阅读器 -> 写作系统闭环

这是更高价值的方向。

未来可以在阅读器内部再拓展出一个轻量写作工作台，或者与主写作模块互通。

可做能力包括：

- 话题仿写
  - 例如围绕某篇 Le Monde 文章做主题仿写
- 句式仿写
  - 把金句抽成模板，让用户做替换训练
- 结构仿写
  - 学习文章如何起承转合，再按同结构写自己的版本
- 观点改写
  - 用不同表达重写原观点
- 论证扩写
  - 给出核心句，让用户补全论据

如果这部分成型，产品就不再只是“输入型工具”，而是“输入 + 输出双驱动工具”。

### 路线三：阅读器 -> 个人知识资产系统

长期看，阅读器可以成为你的“法语知识中台”。

因为它天然适合沉淀这些资产：

- 文章快照
- 高亮
- 笔记
- 表达库
- 句式库
- 论证模板
- 写作素材
- 闪卡来源链路

当这些资产积累起来后，产品价值会出现质变：

- 不是一次性阅读
- 而是形成个人长期可复用语料库

## 六、建议的分阶段路线图

为了避免未来改到失控，建议按下面阶段推进。

### 阶段 1：把当前 Reader MVP 做稳

重点：

- 导入更稳定
- 高亮更稳定
- 清理模式更稳定
- 接口错误提示更清晰
- 保存和重新打开更可靠

目标：

- 达到“日常可用”

### 阶段 2：做强阅读标注系统

重点：

- 高亮颜色语义化
- 标签化
- 结构化笔记
- 高亮筛选与检索

目标：

- 达到“阅读加工可用”

### 阶段 3：接上闪卡系统

重点：

- 词袋工作流
- 批量制卡
- 从卡片回跳原文
- 表达优先级管理

目标：

- 达到“阅读转记忆闭环”

### 阶段 4：接上 AI 辅助

重点：

- 文章大纲
- 摘要
- 金句提炼
- 可制卡表达提取
- 难句解释

目标：

- 达到“阅读增效”

### 阶段 5：接上写作系统

重点：

- 句式仿写
- 观点仿写
- 话题仿写
- 素材引用
- 阅读与写作联动

目标：

- 达到“输入输出一体化”

## 七、我对这个产品方向的判断

我认为这个阅读器真正有潜力的地方，不在于“像一个普通网页阅读器”，而在于它能成为你整个法语学习系统里的中枢模块。

它未来最有竞争力的地方，不是单点功能，而是以下组合能力：

- 阅读时能加工
- 加工后能沉淀
- 沉淀后能制卡
- 制卡后能复习
- 复习后能回跳原文
- 原文中的表达还能继续进入写作训练

一旦这条链路打通，它就不是单个 Reader 页面，而是：

“阅读器 + 语言资产库 + 制卡引擎 + 写作素材库”的联合入口。

## 八、今日结论

今天的 `reader` 工作可以正式收口。

当前状态可以定义为：

- 页面排版已恢复正常
- 核心 MVP 功能已恢复可运行
- 阅读器与后端 API 已重新接通
- 后续产品方向已经明确，不再只是修 bug，而是可以进入体系化演进

下一次继续推进时，建议优先顺序如下：

1. 让高亮 / 笔记系统从“能用”升级到“好用”
2. 让 Reader 与闪卡系统形成真正闭环
3. 接入 AI 做大纲、金句和表达提炼
4. 再进入阅读器与写作系统联动阶段
