# 阅读器方案：开源对比与落地计划（高亮/笔记/汇总 -> 进卡片系统）

记录日期：2026-03-22（Europe/Paris）

## 0. 决策与约束（本文件已按此更新）

1. 路线选择：**内置阅读器**（不依赖外部阅读库同步），但 UI/体验风格**参考 Omnivore**（高亮、侧边栏、快捷键、阅读流畅度）。
2. 初期原文建设：**手动复制粘贴导入**（避免爬取与反爬复杂度）。后期可加 URL 抓取/Reader Mode 抽取作为增强。
3. 长期设备：Surface（触控 + 手写笔记需求明确）；同时为未来 iPad 触控/手写交互留接口与数据模型。
4. 后续拓展：写作模块需要“能引用/打开某篇文章作为参考”，所以阅读器数据要做成稳定 API 与可复用的实体（articles/highlights/notes/ink）。

## 1. 需求汇总（学法语 + 笔记 + 单词入库 + 写作素材）

### A. 文章导入与排版（MVP：粘贴）

- 手动导入：粘贴标题（可选）、来源 URL（可选）、正文（纯文本或 HTML）
- 排版：阅读模式排版（段落、标题、引用、列表、代码块可选），可调整字体大小/行距
- 持久化：保存文章快照（建议存 `content_html`，并保留原始 `content_raw`）
- 复习：能回到文章继续读、继续做高亮与笔记（“二次学习”）

### B. 高亮与笔记（必须）

- 划词/划句/划段高亮；多颜色/标签（例：生词、地道表达、写作素材、论点）
- 每条高亮支持：
  - 笔记（中文/法语都可）
  - 结构化字段（可选）：词条、释义草稿、例句候选
- 高亮定位稳定：至少在**你保存的同一份文章快照**中可稳定复现
- 导出：按文章导出 Markdown（含高亮与笔记）

### C. 词袋（inbox）与制卡联动（必须）

- 从高亮创建“词袋项”（可以是一词/短语/固定搭配）
- 词袋状态：draft/confirmed/rejected（避免“稍后整理”失败）
- 批量制卡：词袋项 + 上下文 + 你的笔记 -> AI 生成卡片草稿 -> 你确认入库
- 反向关联：卡片能跳回来源文章与具体高亮

### D. 手写/触控（Surface 优先，兼顾 iPad）

你在阅读中会做：

- 手写笔记（写在边栏，或直接写在正文旁边）
- 手写高亮/圈画（类似电子书批注）

要求：

- 不和滚动冲突：需要“阅读模式 / 书写模式”切换
- Ink 数据可持久化并可复现（至少在文章快照内）
- 触控设备上交互可用（长按选词、高亮菜单、手写绘制）

### E. 写作模块的复用接口（为后期留好）

- 写作页面可：
  - 搜索文章/高亮/笔记
  - 一键插入某条高亮/例句到写作草稿
  - 在写作侧边栏打开阅读器作为参考（同一篇文章快照）

## 2. GitHub 方案对比（“参考 Omnivore”但内置实现）

结论取向：

- Omnivore：作为产品体验对标（阅读流、侧边栏、高亮与键盘交互），不做直接嵌入。
- Linkwarden / wallabag：更多作为“保存/归档/快照”的参考（如果后期加 URL 抓取与网页快照）。
- Hypothesis：可参考其注释 anchor 思路（DOM Range 定位、注释数据结构），但不引入其服务端体系。

## 3. 技术路径（按阶段交付，确保每阶段都可用）

> 关键原则：先把“粘贴导入 -> 高亮/笔记 -> 词袋 -> 制卡 -> 反向跳转”闭环跑通；URL 抓取、手写层、写作引用都是在稳定数据模型上的增量。

### 阶段 0：数据库与 API 骨架（先做底座）

- 数据库：SQLite（见你的《数据库迭代计划》）
- Flask：新增 `/api/...` 只做文章/高亮/词袋/卡片的 CRUD
- 权限：单用户可先不做登录，但所有数据结构要“可扩展到多用户”（至少字段上留 user_id 的位置）

验收：

- 能创建文章、拉取文章、创建高亮、列出高亮

### 阶段 1（MVP）：粘贴导入 + 阅读排版 + 高亮 + 笔记（不含手写）

前端：

- 新增 `reader.html`：
  - 左侧：正文
  - 右侧 drawer：高亮列表与笔记编辑
  - 顶部工具条：字体大小、行距、显示/隐藏侧栏
- 高亮交互（键盘 + 鼠标）：
  - 选中 -> 弹出浮动菜单：高亮颜色、添加笔记、加入词袋

后端：

- `POST /api/articles`：创建文章（source=paste）
- `GET /api/articles/:id`：拉取文章快照
- `POST /api/articles/:id/highlights`：创建高亮
- `GET /api/articles/:id/highlights`：列出高亮

验收：

- 复制一篇法语文章 -> 粘贴导入 -> 阅读 -> 多处高亮与笔记 -> 关闭页面后再次打开仍完整保留

### 阶段 2：词袋与制卡联动（把“学习闭环”接上）

- `POST /api/vocab/inbox`：从高亮创建词袋项（带 context_sentence + note）
- `POST /api/cards/from-inbox`：批量 AI 生成卡片草稿并存储（先草稿态）
- `POST /api/cards/confirm`：确认入库，进入你的 FSRS 卡片系统
- 关联字段：card 必须带 `source_article_id`、`source_highlight_id`

验收：

- 从高亮创建词袋 -> 一键制卡 -> 在闪卡复习里看到新卡 -> 从卡片跳回来源文章位置

### 阶段 3：Surface 手写（Ink Layer）与触控优化

#### 3.1 交互策略（避免和滚动/选区冲突）

- 模式切换：
  - 阅读模式：正常滚动/选词/高亮
  - 书写模式：禁用文本选择，高亮菜单隐藏，手写落在 ink layer
- Pointer Events：
  - `pointerType === 'pen'` 优先作为“书写输入”
  - touch 与 mouse 在书写模式下也可画（便于 iPad/触屏）

#### 3.2 Ink 数据存储（推荐：矢量笔画）

- 存 stroke 序列（每条 stroke 是点数组）而不是只存截图：
  - 便于缩放与重绘
  - 便于未来导出 SVG/PNG

建议表：

- `ink_strokes`
  - `id`
  - `article_id`
  - `page_version`（文章快照版本号，防止内容变更导致错位）
  - `bounds`（可选：stroke 的包围盒）
  - `points_json`（[{x,y,t,p}]，可选 pressure）
  - `color`, `width`
  - `created_at`

定位策略（先求稳）：

- MVP 先把 ink 定位为“相对文章容器坐标 + 当时的滚动偏移”，只保证在同一快照与同一布局参数下复现。
- 后续可升级为“锚定到某段落/某高亮”的相对定位（更稳，但复杂）。

验收：

- Surface 上用笔写几处批注 -> 关闭 -> 重新打开仍正确显示
- iPad/触控上能切换书写模式并绘制

### 阶段 4：写作模块引用阅读器（复用接口）

新增 API：

- `GET /api/search?q=...`：按文章/高亮/笔记/词条搜索
- `GET /api/articles/:id/summary`：返回可插入写作的引用结构（标题、URL、关键高亮列表）

写作侧交互（最低实现）：

- 写作页面右侧“参考资料”面板：
  - 搜索文章
  - 插入高亮文本到写作区
  - 打开 reader（同一篇文章快照）作为参考

验收：

- 写作页能一键引用你之前高亮的例句/表达

### 阶段 5：URL 抓取与 Reader Mode（增强项，不阻塞前期）

当粘贴导入稳定后，再加：

- `POST /api/articles/import-url`：输入 URL
- 后端抓取 HTML -> Readability/trafilatura 抽取 -> 存快照
- 对付费墙/JS 渲染站点：必要时再引入 Playwright（单独评估）

## 4. 数据模型草案（为手写与写作复用预留）

最小闭环（建议先实现这些表）：

- `articles`
  - `id`, `title`, `source_url`, `content_raw`, `content_html`, `created_at`, `updated_at`, `version`
- `highlights`
  - `id`, `article_id`, `quote_text`, `start_path`, `start_offset`, `end_path`, `end_offset`, `color`, `note`, `created_at`
- `vocab_inbox`
  - `id`, `article_id`, `highlight_id`, `expression`, `context_sentence`, `user_note`, `status`, `created_at`
- `cards`（你的卡片系统）
  - 增加：`source_article_id`, `source_highlight_id`
- `ink_strokes`（阶段 3 再做）

## 5. 前端实现建议（仍保持 Vanilla JS 的可控范围）

### Reader 排版

- 内容建议存为“已清洗的 HTML”，渲染时应用固定排版 CSS（类似 Omnivore 的阅读体验）
- 粘贴时的清洗：
  - 允许最常见的标签：p, h1-h3, ul/ol/li, blockquote, em/strong, a
  - 去掉 style/script/iframe

### 高亮实现（MVP）

- selection -> range
- 计算并存储 anchor（path + offset）
- 渲染时用 `<mark data-highlight-id="...">` 包裹
- 点击 mark 打开侧栏，编辑笔记/颜色/加入词袋

### 手写层（阶段 3）

- 在正文容器上方叠一个透明 canvas
- 书写模式下捕获 pointer 事件并绘制
- 结束 stroke 时把点数组 POST 到后端保存
