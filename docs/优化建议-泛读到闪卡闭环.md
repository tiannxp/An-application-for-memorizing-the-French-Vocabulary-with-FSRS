# 优化建议：打通「泛读积累」与「精记词汇」

记录日期：2026-03-21（Europe/Paris）

## 核心想法的价值

目标是做一个完整的输入到沉淀再到输出的闭环：

阅读原文 -> 一键取词 -> 阅读后整理 -> 制卡记忆 -> 写作/口语素材库

现有工具通常是割裂的：阅读工具负责“看”，Anki 负责“背”，上下文与素材沉淀很难自然串起来。把链路打通对自用学习效率很有价值。

## 需要想清楚的关键问题

### 1) “取词”发生在哪里

这是体验与实现复杂度的最大分叉点：

- 方案 A：浏览器插件
  - 优点：在原网站上直接划词，体验最自然
  - 缺点：需要开发 Chrome Extension；与后端通信、鉴权、跨域与发布都有额外成本
- 方案 B：内置阅读器（推荐优先）
  - 优点：文章导入到自己的 app 内渲染，取词/高亮/定位完全可控；与现有 Flask/本地数据链路更契合
  - 缺点：多一步“导入/抓取”的动作
- 方案 C：剪贴板监听
  - 优点：实现最轻量
  - 缺点：体验粗糙，误触发与上下文丢失较多

结合当前技术栈与“抓取到自己的阅读库”的方向，建议优先走方案 B。

### 2) “整理”是否会变成拖延入口

“阅读结束后再整理”很容易被跳过。建议把整理摩擦降到最低：

- 取词时先进入“待处理词袋”（inbox）
- AI 后台静默生成卡片草稿
- 用户整理阶段只做：确认/微调/删除，而不是从零填写

### 3) 阅读库与词库如何建模

尽量保留“这个词是在哪篇文章的哪句话里学到的”上下文，既利于记忆，也能支持素材库/写作复用：

- Article（文章）
- Highlight（划线/取词片段，包含位置信息）
- Card（闪卡，可能尚未生成）

关系示意：

Article
  highlights[]
    -> card?  (可为空，表示还在词袋里未制卡)

## 建议分两期做（可落地）

### 第一期（最小可用）

只做：内置阅读器 + 取词暂存 + 批量制卡。

- URL 粘贴导入
- 后端抓正文：`trafilatura` / `newspaper3k`（可先从简单站点开始）
- 前端渲染正文，支持划词/高亮/暂存到“词袋”
- 阅读结束后一键批量送 AI 生成卡片草稿，进入确认流程

### 第二期（素材库）

围绕“输出复用”做增量：

- 文章标签/主题归档
- 高亮片段可写笔记
- 例句片段与既有词卡关联，形成可搜索的写作素材库

## 潜在风险（提前规避）

订阅/付费媒体与强反爬站点可能出现：

- 直接 `requests` 抓取拿到空内容（JS 渲染后才出现）
- 反爬/频率限制
- 登录态/订阅态依赖

必要时需要无头浏览器（如 `playwright`）抓取，会显著增加复杂度。建议先列出常读站点做可抓取性验证，再决定技术路线。

## 待决策清单

- 取词入口最终是否需要浏览器插件（一期是否完全不做）
- 文章导入：只支持 URL，还是也支持粘贴全文/上传文件
- “词袋”整理的 UI 形态：列表确认 vs. 卡片逐个确认
- 数据保存位置：先 JSON/SQLite，再回写 Excel（或逐步淘汰 Excel）
