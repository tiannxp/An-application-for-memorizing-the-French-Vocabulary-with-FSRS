# 数据库迁移实施记录与使用说明

日期：2026-03-27

## 1. 这次改造完成了什么

这次改造的核心目标，是把原先分散在多个位置的数据源，收束到一个统一的 SQLite 数据库里。

改造前，系统主链路大致是：

- AI/后端写入 `french_app_data.xlsx`
- 再由脚本构建 `database_structured.json`
- 再由脚本构建 `data_frontend.json`
- 前端复习时把状态存在 `localStorage`
- 用户手动点“同步”后，再把 `localStorage` 的状态回写 Excel
- Reader 则另外使用 `database/reader.sqlite3`

改造后，系统主链路变成：

- 闪卡、复习状态、阅读器文章、高亮、词袋，都进入同一个库：`database/app.sqlite3`
- 前端复习页面直接读 API，不再依赖 `data_frontend.json`
- 复习打分后直接写回 SQLite，不再先写 `localStorage` 再手动同步
- Reader 不再走独立事实源，和闪卡共享同一个库
- Excel 现在降级为“导出备份格式”，不再是运行时主存储

## 3. 这次是怎么实现的

### 第一步：先建立统一数据库模型

先没有直接碰前端，而是先把后端的数据底座建好。

我在 `database_store.py` 中建立了统一主库 `database/app.sqlite3` 的 schema，包括：

- `cards`
- `tags`
- `card_tags`
- `review_logs`
- `articles`
- `highlights`
- `vocab_inbox`
- `ink_strokes`

这样做的目的，是保证后面所有改造都围绕一个稳定的数据模型来展开，而不是边改边继续扩散新的数据源。

### 第二步：实现历史数据导入

然后实现了两个导入方向：

1. Excel -> SQLite
2. `reader.sqlite3` -> SQLite

这一步做完以后，主库就不是空壳了，而是可以真正承接原来的闪卡和 Reader 数据。

### 第三步：把 Flask API 改为直接面向主库

原本 `app.py` 同时混杂了：

- Excel 写入
- JSON 重建
- Reader 独立 SQLite
- 前端同步接口

我把这些职责改成：

- 统一由 `database_store.py` 负责存取
- `app.py` 只负责 API 路由和页面入口
- 新增闪卡 API，让复习链路直接走数据库

### 第四步：把前端从 JSON/localStorage 切到 API

这一步是用户体验变化最大的地方。

原来前端会：

- 先加载 `data_frontend.json`
- 再用 `localStorage` 合并学习进度
- 用户手动点“同步”后，才有机会回写后端

现在改成：

- 页面直接拉 `/api/cards/due`
- 用户评分后立刻请求 `/api/cards/<id>/review`
- 后端直接落库
- 再读 `/api/review/today` 或下一轮 `/api/cards/due` 时，就是最新状态

## 4. 过程中走过的弯路和实验

这部分单独写出来，方便之后回看，也方便以后继续迭代时少踩坑。

### 4.1 一开始尝试大块补丁一次性写入数据库层

我最开始尝试把 `database_store.py` 一次性用一个大 patch 写进去，但在 Windows 沙箱环境里撞到了命令长度限制，报了 `CreateProcessAsUserW failed: 206`。

后来的处理方式是：

- 把 `database_store.py` 拆成多段 patch 分次写入

这不是逻辑问题，是工具和系统边界问题。

### 4.2 默认 Python 命令不可用

我最开始用 `python` 和 `py` 跑语法检查和导入，结果：

- `python` 指向了一个不可直接使用的 stub
- `py` 提示 `No installed Python found!`

后来我改成显式使用本机路径：

- `C:\Users\surface\AppData\Local\Programs\Python\Python312\python.exe`
- `C:\Users\surface\miniconda3\python.exe`

### 4.3 Python 312 环境缺少 pandas/openpyxl

虽然 `Python312` 可以做语法检查，但跑导入脚本时失败，因为没有 `pandas` / `openpyxl`。

后来改用：

- `C:\Users\surface\miniconda3\python.exe`

这个环境能顺利完成 Excel 导入。

### 4.4 导入脚本第一次看起来“像失败了”，其实是控制台编码问题

第一次跑 `import_to_sqlite.py` 时，脚本最后输出 JSON 结果时触发了 `UnicodeEncodeError`，表面上像“导入失败”。

后来排查发现：

- 数据其实已经成功写进了 `database/app.sqlite3`
- 报错发生在最后的 `print(json.dumps(...))`
- 本质是 Windows 控制台的编码问题

后来的解决方式是：

- 设置 `PYTHONIOENCODING=utf-8` 后重新运行

### 4.5 原来的前端状态假设嵌得太深，局部修补反而更危险

我原本尝试只局部替换 `script.js` 里的几段逻辑，把 `data_frontend.json` 和 `localStorage` 的部分改掉。

但很快发现：

- 原前端流程的很多函数默认“本地存储就是主库”
- 局部修补容易留下半新半旧的逻辑

所以后来改成：

- 直接重写 `script.js` 的数据流部分
- 保留页面结构和主要交互
- 把数据来源彻底改成 API

这条路反而更稳。

## 7. 现在和以前相比，使用上有什么不同

这是对你最重要的部分。

### 以前

- 闪卡内容来自 `data_frontend.json`
- 学习进度主要存在浏览器 `localStorage`
- 需要手动点“同步”才能把进度回写
- Excel 是实际主库
- Reader 用的是另一套独立 SQLite

### 现在

- 闪卡内容直接来自 SQLite + API
- 学习进度直接写入 SQLite
- 不再需要把 `localStorage` 当主数据库
- 不需要依赖 `data_frontend.json` 才能复习
- Reader 和闪卡共享同一个数据库
- Excel 只作为导出备份，不是运行时主存储

### 对你实际操作的影响

最直接的变化有三点：

1. 你复习后不需要再担心“有没有同步到 Excel”
2. 浏览器本地缓存不是主数据源了，换浏览器/清缓存不再等于丢主进度
3. Reader 和闪卡之间的数据关系以后更容易打通，比如：
   - 从高亮生成卡
   - 从卡跳回文章
   - 给写作模块复用
